# 视频通信实施方案

你这个设计思路在整体方向上是对的，但要结合现有架构（Gate/Chat/Status/Varify/QChat/TurnServer + 已部署的 coturn）做一次“体系化整合”，保证协议清晰、并发安全、实现难度可控。

下面给一个分三层的实施方案：

1. 整体架构与数据流（先说“怎么走”）
2. 协议设计：客户端↔ChatServer、ChatServer↔ChatServer、客户端↔WebRTC/TurnServer（再说“说什么”）
3. 并发与可靠性要点（最后说“怎么保证正确性和并发性”）

---

## 1. 整体架构与数据流

### 1.1 角色

- `QChat`：Qt 客户端，新增 `视频通话窗口` + `信令/控制逻辑`
- `ChatServer/ChatServer2`：原有聊天 TCP 服务器，新增“视频通话控制信令”处理（不负责媒体，只负责控制）
- `StatusServer`：保持不动，仅负责“好友在哪个 ChatServer，上线/下线状态”
- `TurnServer`（你写的 Node WebSocket 信令服务器）：只做 WebRTC Peer 之间的房间匹配 + offer/answer/ice 透传
- `coturn`：纯粹的 TURN/STUN，供 WebRTC 媒体通道打洞/中继用
- 其他服务（Gate/Varify/Resource）：基本不用动

### 1.2 呼叫流程（单机 ChatServer 上的简单情况）

以“用户 A 给用户 B 发起视频通话”为例：

1. A 客户端在“聊天窗口”点击“视频通话”
2. A 客户端通过现有 TCP 连接向自己的 `ChatServer` 发送：`CALL_INVITE_REQ`（自定义一个新消息类型）
3. `ChatServer` 通过 `StatusServer` 查 B 是否在线、在哪个 `ChatServer`：
    - 若 B 不在线：直接给 A 回复 `CALL_INVITE_RSP{ error = 对方不在线 }`
    - 若 B 在线且在**同一台** ChatServer：直接通过当前 TCP 连接转发 `CALL_INCOMING_NOTIFY` 给 B
4. B 客户端收到 `CALL_INCOMING_NOTIFY`：弹出“来电窗口”（带 拒绝/接受 按钮）
5. B 选择：
    - 拒绝：B 发 `CALL_REJECT_REQ` → ChatServer → 通知 A：`CALL_REJECT_NOTIFY`，然后双方结束这次请求
    - 接受：B 发 `CALL_ACCEPT_REQ` → ChatServer → 通知 A：`CALL_ACCEPT_NOTIFY{ room_id, turn_server_url, ice_servers }`
6. 双方客户端在收到“接受”相关消息时，各自完成两件事：
   
    1）去连 TurnServer（你的 Node WebSocket `TurnServer/server.js`）并 `join(room_id)`
    
    2）在本地创建 WebRTC PeerConnection，配上 `iceServers`（指向你的 coturn）
    
7. 当某一端点击“挂断”：
    - 客户端发 `CALL_HANGUP_REQ` → ChatServer → 通知对端 `CALL_HANGUP_NOTIFY`
    - 双方客户端关闭 WebRTC Peer 和 WebSocket/房间，UI 界面关闭

### 1.3 跨 ChatServer 情况（分布式）

当 A 与 B 不在同一台 ChatServer 时：

1. A→ChatServerA：`CALL_INVITE_REQ`
2. ChatServerA：
    - 向 `StatusServer` 请求 B 当前所在 `ChatServerId`（这个在你现有系统里已经有类似逻辑）
    - 通过现有 gRPC 通道，向 `ChatServerB` 发 `CALL_REMOTE_INVITE_REQ`（server-to-server 协议）
3. ChatServerB：
    - 找到 B 的连接（你们已经维护 uid→连接 的映射）
    - 给 B 发送 `CALL_INCOMING_NOTIFY`
4. B 接受/拒绝时：
    - B→ChatServerB： `CALL_ACCEPT_REQ / CALL_REJECT_REQ`
    - ChatServerB→ChatServerA：gRPC `CALL_REMOTE_ACCEPT/REJECT_RSP`
    - ChatServerA→A：`CALL_ACCEPT_NOTIFY/CALL_REJECT_NOTIFY`
5. room_id 的生成建议由**发起侧 ChatServer** 统一生成（如 `chatId_timestamp_rand`），再通过 gRPC 带到对端：
    - 这样双方拿到的是同一个 `room_id`，连接 TurnServer 时直接用

## 2. 协议设计

下面只给出“字段级别的协议定义思路”，不展开具体 C++ 代码实现（你可以按现有项目 `message.proto` 的风格加上去）。

### 2.1 客户端 ↔ ChatServer 的业务消息

在你现在 `TcpMgr` 注册的各种 `ReqId` 上，再扩展一组“视频通话控制”：

### 2.1.1 发起呼叫

- 名称：`CALL_INVITE_REQ`
- 方向：客户端 → ChatServer
- 字段（JSON 或 Protobuf 按你现有格式）：
    - `caller_uid`
    - `caller_nick`
    - `callee_uid`
    - `chat_type`（可选：单聊/群聊，先只做单聊）
- 响应：`CALL_INVITE_RSP`
    - `error`（0=成功，非0表示失败，如对方不在线、自己正在通话中等）
    - `call_id`（可选，用于多设备/以后扩展）

如果成功 + 在线，服务器不等 B 接受就先返回“成功（已发送来电通知）”。

### 2.1.2 来电通知

- 名称：`CALL_INCOMING_NOTIFY`
- 方向：ChatServer → 客户端
- 字段：
    - `caller_uid`
    - `caller_nick`
    - `call_id`

### 2.1.3 接受/拒绝来电

- 接受：`CALL_ACCEPT_REQ`
    - `call_id`
    - `room_id`
    - `turn_ws_url`
    - `ice_servers`
- 拒绝：`CALL_REJECT_REQ`
    - `call_id`
    - `reason`（可选：忙线、拒绝、无响应等）
- 服务器给**各自**返回：
    - `CALL_ACCEPT_RSP` / `CALL_REJECT_RSP` 给自己（保证请求成功）
    - `CALL_ACCEPT_NOTIFY` / `CALL_REJECT_NOTIFY` 给对方

`CALL_ACCEPT_NOTIFY` 字段设计：

```json
{
  "call_id": "...",
  "room_id": "1234",
  "turn_ws_url": "wss://your-signaling.domain/turn",
  "ice_servers": [
    {
      "urls": ["turn:coturn.domain:3478"],
      "username": "xxx",
      "credential": "yyy"
    },
    {
      "urls": ["stun:stun.l.google.com:19302"]
    }
  ]
}
```

说明：

- `room_id` 用来连接你的 Node `TurnServer`，即 `join(room_id)`（这里room_id直接用发起方caller的uid作为房间号，确保房间号唯一）
- `turn_ws_url` 是 Node 信令服务器地址
- `ice_servers` 直接作为 Qt WebRTC/第三方库的 ICE 配置（客户端不需要知道 coturn 细节）

`CALL_REJECT_NOTIFY` 字段：

```json
{
  "call_id": "...",
  "reason": "rejected|busy|timeout|network_error"
}
```

### 2.1.4 挂断

- 请求：`CALL_HANGUP_REQ`
    - `call_id`
    - `by`（caller/callee，服务器也可自行判断）
- 返回自己：`CALL_HANGUP_RSP`
- 通知对方：`CALL_HANGUP_NOTIFY`

客户端逻辑：

- 收到 `HANGUP_NOTIFY` 时关闭本地 WebRTC、关闭视频 UI
- 自己点挂断时：
    - 先给 WebRTC peer 发 `close`，关闭音视频
    - 再发 `CALL_HANGUP_REQ` 给 ChatServer

### 2.2 ChatServer ↔ ChatServer（gRPC）协议

在你的 `ChatServer/message.proto` 基础上，增加一个 `VideoCallService`（伪代码）：

```
service VideoCallService {
  rpc RemoteInvite(RemoteCallInviteReq) returns (RemoteCallInviteRsp);
  rpc RemoteAccept(RemoteCallAcceptReq) returns (RemoteCallAcceptRsp);
  rpc RemoteReject(RemoteCallRejectReq) returns (RemoteCallRejectRsp);
  rpc RemoteHangup(RemoteCallHangupReq) returns (RemoteCallHangupRsp);
}

message RemoteCallInviteReq {
  int64 caller_uid = 1;
  int64 callee_uid = 2;
  string call_id = 3;
}

message RemoteCallInviteRsp {
  int32 error = 1;
}

message RemoteCallAcceptReq {
  string call_id = 1;
  int64 caller_uid = 2;
  int64 callee_uid = 3;
  string room_id = 4;
}

message RemoteCallAcceptRsp {
  int32 error = 1;
}

message RemoteCallRejectReq {
  string call_id = 1;
  int64 caller_uid = 2;
  int64 callee_uid = 3;
  string reason = 4;
}

message RemoteCallHangupReq {
  string call_id = 1;
  int64 caller_uid = 2;
  int64 callee_uid = 3;
}
```

处理逻辑：

- `RemoteInvite`：在目标 ChatServer 上找到 callee 的连接，发 `CALL_INCOMING_NOTIFY`
- `RemoteAccept`：目标 ChatServer 把 `CALL_ACCEPT_NOTIFY{ room_id, ice_servers }` 发给 caller（通过 gRPC 回源）
- `RemoteReject/RemoteHangup` 同理

你已经在 `ChatServer` 系统里有 gRPC 通信能力，这里只是增加一个 service。

### 2.3 WebRTC 信令层（TurnServer）

你当前 `TurnServer/server.js` 已经支持的消息类型：

- `join`：进入房间，如果满 2 人就给双方发 `ready{ isInitiator }`
- `offer` / `answer` / `ice`：透传给房间内另一个人
- `leave`：离开房间

这里可以做两个小改进以增强**并发和正确性**：

1. 在 `join` 消息体内加入 `call_id`、`uid`：

```json
{ "type": "join", "roomId": "xxx", "uid": 1001, "call_id": "abc" }
```

在 `ws` 对象上记录：

```jsx
ws.roomId = roomId;
   ws.uid = msg.uid;
   ws.callId = msg.call_id;
```

这样后续可以做一些“断线重连/异常调试”等逻辑（比如同一个 uid 重复 join 同一 room 时可以踢掉旧连接）。

1. 加上**心跳/超时清理**（保证房间不会泄露）：
    - 在 `connection` 里给每个 `ws` 设置一个心跳定时器，每 30s 检测最后一次 ping/pong 时间
    - 若超时（如 90s 无心跳），调用 `removeFromRoom(ws)` 并 `ws.terminate()`
2. 如果你考虑到**一对多（以后可能要屏幕分享给多个观众）**，可以先保留 `rooms: Map<roomId, Set<ws>>`，逻辑上只允许“caller/callee” 两个角色同时存在。现在的实现已经做了 `set.size >= 2` 检查，这是正确的。

## 3. 客户端（Qt）实现建议

不展开具体 C++ 代码，只说结构和职责。

### 3.1 UI 结构

在 `QChat` 工程中新增：

- `videocallwindow.h/.cpp/.ui`：
    - 显示本地视频画面（例如 `QVideoWidget` 或你用的 WebRTC Qt 封装）
    - 显示远端视频画面
    - 控制按钮：挂断、静音、关视频、切换摄像头等等（先实现挂断）
- 增加一个“来电弹窗” UI：
    - 内容：来电头像 + 昵称 + “接受/拒绝” 按钮

在当前的聊天窗口 `ChatDialog` 或类似类中增加一个“视频通话按钮”，点击后调用 `TcpMgr` 新增的接口触发 `CALL_INVITE_REQ`。

### 3.2 `TcpMgr` 中增加消息处理

在你给出的 `tcpmgr.cpp` 里面已经有各种 `_handlers.insert(...)`，你可以再增加：

```cpp
_handlers.insert(ID_CALL_INVITE_RSP, [this](ReqId id, int len, QByteArray data) { ... });
_handlers.insert(ID_CALL_INCOMING_NOTIFY, [this](ReqId, int, QByteArray) { ... });
_handlers.insert(ID_CALL_ACCEPT_NOTIFY, ...);
_handlers.insert(ID_CALL_REJECT_NOTIFY, ...);
_handlers.insert(ID_CALL_HANGUP_NOTIFY, ...);
```

这些 handler 主要做两件事：

1. 把 JSON 解析为 `VideoCallEvent` 对象
2. 通过 Qt 信号/槽发给 UI 层：
    - `sigIncomingCall(caller_uid, call_id)`
    - `sigCallAccepted(room_id, turn_ws_url, ice_servers)`
    - `sigCallRejected(reason)`
    - `sigCallHangup()`

UI 层接到信号，再创建/销毁 `VideoCallWindow`，以及驱动 WebRTC 模块。

### 3.3 WebRTC 客户端模块

Qt 里推荐包一层 `VideoCallManager`：

- 对外接口：
    - `startCall(roomId, turnWsUrl, iceServers)`
    - `acceptCall(roomId, turnWsUrl, iceServers)`
    - `hangup()`
- 内部动作：
    1. 用 Qt 的 WebSocket 客户端连你的 `TurnServer`：
        - 发送 `{type:"join", roomId, uid, call_id}`
        - 收到 `ready{isInitiator}` 后：
            - 若 `isInitiator==true`：创建 offer，发 `offer` 消息给 TurnServer
            - 若 `isInitiator==false`：等待 `offer`，再生成 answer
        - IceCandidate 收发映射到 `ice` 消息
    2. 用 WebRTC 库（可选：Qt WebRTC wrapper、libwebrtc 封装库等）
        - 配置 ICE：用服务器下发的 `ice_servers`
        - 设置本地摄像头/麦克风
        - 绑定视频 track 到 UI 里的 `QVideoWidget` 或其他渲染组件
- 当挂断或通话中断时：
    - 关闭 PeerConnection
    - 发送 `leave` 到 TurnServer
    - 关闭 WebSocket

## 4. 并发与正确性要点

### 4.1 避免“多设备冲突”和“并发呼叫”问题

在 `ChatServer` 端增加一个“用户通话状态表”（内存即可）：

- 结构：`unordered_map<uid, CallSession>`
    - `CallSession` 内含：`call_id, peer_uid, start_time, state` 等

当收到 `CALL_INVITE_REQ` 时：

1. 如果 `caller` 或 `callee` 任一方已经处于 `InCall` 或 `Ringing` 状态，则返回错误码：`ERR_BUSY`，并给 UI 提示“对方正在通话中”。
2. 成功进入“呼叫中（Ringing）”状态后，需要在超时（如 30 秒）后自动清理：
    - 如果 B 超时不响应，ChatServer 自动给 A 发 `CALL_REJECT_NOTIFY{ reason="timeout" }`，并清除状态。

挂断/拒绝/通话异常断开后都要：

- 更新状态为 `Idle`
- 删除 `CallSession`

### 4.2 分布式一致性

当涉及到 ChatServerA / ChatServerB 时：

- 状态表可以**每台 ChatServer 本地维护**即可，关键是“谁是 owner”：
    - 建议：**发起侧 ChatServer** 是 `call_id` 的 owner：
        - 生成 `call_id` & `room_id`
        - 对本地和远端的 `caller_uid`、`callee_uid` 状态进行驱动
    - 远端 ChatServer 只做“透传通知”和“确认”，自身可以维护“我这台 server 上的 uid 正在与哪个 call_id 通话”，但不做全局裁决

这样可以减少跨节点状态同步带来的复杂性。

### 4.3 TurnServer 并发

你现在的 `TurnServer/server.js` 由于是 Node 单线程 + `rooms:Map<roomId, Set<ws>>`，只要：

- 不在消息回调里做阻塞操作（现在没有）
- 对 `rooms` 只在主线程访问

那线程安全问题基本没有。需要注意：

- 大量长连接下要配置操作系统的 fd 上限和 Node 进程的内存限制
- 可以为 TurnServer 单独部署一台云服务器，配 l4 负载（后面可再拆分）

## 5. 实施顺序建议

避免“一上来全做完”，建议按这种顺序分步：

1. **先在单机 ChatServer 上打通最小闭环**：
    - 单个 ChatServer、单机部署 TurnServer + coturn
    - 完成：A 与 B 在同一个 ChatServer 上的视频通话（不涉及 gRPC 跨服）
2. **再扩展到跨 ChatServer**：
    - 在现有 gRPC 工程中新增 VideoCallService 协议和实现
    - 通过 StatusServer 确定对端 ChatServerId，并调用远程接口
    - 验证 A、B 分别登录两台 ChatServer 时依然可以正常视频通话
3. **最后处理异常/完善体验**：
    - 超时自动拒接
    - 对方网络断开自动挂断
    - 多端登录时的策略（比如只向在线的“手机端”推送视频通话）
