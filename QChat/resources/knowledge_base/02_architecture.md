## 各服务器组件详解

### 1. GateServer（网关服务器）

**职责**：
- 系统的 HTTP 入口点
- 处理用户注册、登录、密码重置等 API 请求
- 与下游服务（StatusServer）通信获取聊天服务器地址

**技术栈**：
- C++、Boost.Asio、JSON

**端口**：13419

**核心流程**：
1. 用户登录请求 -> GateServer
2. GateServer 调用 StatusServer gRPC 获取最优 ChatServer
3. 返回 ChatServer 地址 + Token 给客户端
4. 客户端直接连接 ChatServer

### 2. ChatServer / ChatServer2（聊天服务器）

**职责**：
- 处理客户端 TCP 长连接
- 实现即时消息传递
- 好友关系管理
- 聊天记录存储

**技术栈**：
- C++、Boost.Asio、gRPC

**端口**：
- TCP: 8090 (ChatServer), 8091 (ChatServer2)
- gRPC: 50055 (ChatServer), 50056 (ChatServer2)

**核心功能**：
- 客户端连接管理（心跳检测、异常断开处理）
- 消息路由（根据 thread_id 找到目标用户连接）
- 消息持久化（写入 MySQL）
- 消息推送（推送到在线用户，离线用户登录后拉取）

### 3. StatusServer（状态服务器）

**职责**：
- 负载均衡（选择最空闲的聊天服务器）
- 用户 Token 管理
- 会话状态验证
- 服务器注册与发现

**技术栈**：
- C++、gRPC

**端口**：50052

**负载均衡算法**：
- 根据各 ChatServer 的当前连接数
- 选择连接数最少的服务器
- 支持动态上下线

### 4. VarifyServer（验证服务器）

**职责**：
- 邮件验证码服务
- 验证码生成、发送、验证

**技术栈**：
- Node.js、gRPC、Redis

**端口**：50051

**流程**：
1. 用户请求验证码
2. 生成 6 位随机码
3. 存入 Redis（5 分钟过期）
4. 发送邮件
5. 用户提交验证码，从 Redis 验证

### 信号槽架构

客户端大量使用 Qt 信号槽机制进行模块间通信：

```cpp
// TcpMgr 信号
sig_con_success(bool)           // TCP 连接成功
sig_text_chat_msg(list)         // 收到文本消息
sig_img_chat_msg(data)          // 收到图片消息
sig_call_incoming(uid, call_id) // 视频通话邀请

// UserMgr 信号
sig_friend_status_changed(uid, online)  // 好友上线/下线
sig_group_chat_created(thread_id, name) // 群聊创建
```

## 数据流

### 登录流程

```
1. 客户端 -> GateServer: POST /login {email, pwd}
2. GateServer -> StatusServer: gRPC GetChatServer()
3. StatusServer -> GateServer: {chat_server_host, chat_server_port}
4. GateServer -> 客户端: {token, uid, chat_server_info}
5. 客户端 -> ChatServer: TCP 连接 + ID_CHAT_LOGIN
6. ChatServer -> 客户端: ID_CHAT_LOGIN_RSP
```

### 发送消息流程

```
1. 客户端 A -> ChatServer: ID_TEXT_CHAT_MSG_REQ
2. ChatServer: 写入 MySQL
3. ChatServer -> 客户端 B: ID_NOTIFY_TEXT_CHAT_MSG_REQ (如果在线)
4. ChatServer -> 客户端 A: ID_TEXT_CHAT_MSG_RSP (确认)
```

### 视频通话流程

```
1. 客户端 A -> ChatServer: ID_CALL_INVITE_REQ {call_id, target_uid}
2. ChatServer -> 客户端 B: ID_CALL_INCOMING_NOTIFY
3. 客户端 B -> ChatServer: ID_CALL_ACCEPT_REQ
4. ChatServer -> TurnServer: 创建房间
5. ChatServer -> 客户端 A/B: ID_CALL_ACCEPT_NOTIFY {room_id, turn_ws_url, ice_servers}
6. 客户端 A/B: 建立 WebRTC P2P 连接（通过 TURN）
```

## 扩展性设计

### 水平扩展

- **ChatServer**：可部署多个实例，StatusServer 自动负载均衡
- **GateServer**：可前置 Nginx 反向代理
- **数据库**：可读写分离、分库分表

### 功能扩展

- **插件系统**：AI Agent 工具注册机制
- **知识库**：支持动态添加文档
- **新消息类型**：通过 MsgType 枚举扩展

## 安全设计

1. **Token 机制**：登录后获取 Token，后续请求携带
2. **异地登录检测**：同一账号多处登录会踢掉旧连接
3. **密码加密**：客户端 XOR 加密传输
4. **验证码**：邮箱验证码防暴力破解
