## 第二轮修复：崩溃与信令流程 Bug（来自实际运行日志分析）

运行日志显示：对方同意视频邀请后双端直接卡死崩溃。定位到以下 4 个新 Bug：

### Bug A【致命-已修复】：`memset(streamconfig)` + `init(peer)` 重复调用导致 crash
- **位置**：`YangRtcWrapper.cpp` 构造函数
- **原因**：上一轮把 `memset(peer,0)` 改成了 `memset(&peer->streamconfig,0)`，但 `yang_create_peerConnection` 内部会填充 `streamconfig.sslCallback` 等内部 DTLS 函数指针；`memset(streamconfig)` 把这些指针清零后，再调用 `m_peerConnection.init(peer)` 就解引用 NULL，导致立即 crash
- **修复**：完全删除 `memset(streamconfig)` 和 `init(peer)` 调用；构造函数只做"追加写应用层字段"

### Bug B【严重-已修复】：`connectSfuServer` 从未调用 → 连接永远无法建立
- **位置**：`YangRtcWrapper.cpp` 中 `connectToServer()` 实现
- **原因**：metartc P2P 模式下，SDP 协商完成后必须调用 `connectSfuServer` 才能触发底层 ICE/DTLS 握手，原来 `connectToServer()` 直接返回 `true` 什么都没做
- **修复**：实现 `connectToServer()` 调用 `m_peerConnection.connectSfuServer`；在 `setLocalDescription`（Answerer 路径）和 `setRemoteDescription`（Offerer 收到 answer 路径）完成后分别按时机触发

### Bug C【严重-已修复】：ChatServer TurnServer WsUrl = `ws://127.0.0.1:3000` → 客户端连本机
- **位置**：`ChatServer/config.ini` + `LogicSystem.cpp` fallback 地址
- **修复**：改为 `ws://47.113.108.95:3000`（信令服务器公网 IP）

### Bug D【重要-已修复】：`startLocalVideo()` 在 Connected 回调里重复调用 → 摄像头二次初始化崩溃
- **位置**：`videocallmanager.cpp` 的 `initMetartc()` 内部 Connected 分支
- **修复**：删除 Connected 分支里的 `startLocalVideo()` 调用；摄像头只在 `handleAcceptCall/handleCallAccept` 里初始化一次

正确的 YangPeerConnection 初始化顺序（严禁改动）：
1. `memset(&m_peerConnection, 0)` → 2. `yang_create_peerConnection` → 3. 只写应用层字段（avinfo、recvCallback）
绝不在步骤2之后对 peer/streamconfig 整体 memset，绝不手动调用 init(peer)

## 第三轮修复：connectSfuServer 是 P2P 模式下的空指针（来自用户反馈卡在 configureIceServers 后）

### Bug E【致命-已修复】：`connectSfuServer` 在 P2P 模式下是 NULL → sigError → endCall
- **发现过程**：用户反馈上一轮修复后流程卡在 `configureIceServers` 完成就停了
- **根本原因**：上一轮为了修 "SDP 交换后 ICE 不启动" 问题，在 `connectToServer()` 里调用了 `m_peerConnection.connectSfuServer`。但该函数指针的注释是 `//srs zlm`，在 `Yang_Server_P2p` 模式下 `yang_create_peerConnection` 不填充它（NULL）。于是 `if (!connectSfuServer)` 成立 → `emit sigError` → `endCall()`，通话立即结束
- **P2P 模式正确行为**：ICE/DTLS 握手由 metartc **内部**在 `setRemoteDescription` 解析到对端 ICE 候选地址后自动启动，外部不需要也不能调用 `connectSfuServer`
- **修复**：`connectToServer()` 改回直接 return true；`setLocalDescription`/`setRemoteDescription` 里移除 `connectToServer()` 调用；改为双端 SDP 都就位后启动 `m_connectTimer` 轮询 `isConnected(peer)`
- **关键结论**：metartc YangPeerConnection 各函数指针的含义严格区分模式，P2P / SRS / ZLM / WHIP-WHEP 模式分别填充不同的函数指针，跨模式调用必然是 NULL

## 第四轮修复：yang_create_peerConnection 内部自动调用 init()（阅读 metartc7 源码后的根本修复）

### 根本问题发现
阅读了 `metartc7/libmetartccore7/src/yangrtc/YangPeerConnection.c` 源码，发现：
- `yang_create_peerConnection()` 最后一行（206行）自动调用 `yang_pc_init(&peerconn->peer)`
- `yang_pc_init()` 会读取 `peer->avinfo` 和 `peer->streamconfig` 传给 `yang_create_rtcConnection()`
- `yang_pc_init()` 有 `if(peer->conn!=NULL) return` 保护，重复调用会被直接跳过
- 之前所有修复都是在 **先 `yang_create_peerConnection` 再设 avinfo** 的错误顺序上打补丁，导致内部 init 用的是 NULL avinfo

### Bug F【致命-已修复】：avinfo/streamconfig 必须在 yang_create_peerConnection **之前**设置
- **参考**：`YangPeerConnection2` 构造函数（先设 avinfo/streamconfig → 再调 yang_create_peerConnection）；`YangRtcReceive::init()` demo（同上顺序）
- **修复**：重构构造函数初始化顺序为 memset → 设 avinfo → 设 streamconfig（含 recvCallback） → `yang_create_peerConnection`
- **删除**：`initRtcConnection()` 中的手动 `init(peer)` 调用（无效，会被 yang_pc_init 跳过）

### Bug G【重要-已修复】：缺少 addTransceiver 调用
- **参考**：demo 中 `YangRtcReceive::init()` 和 `YangRtcPublish::init()` 都在 addAudioTrack/addVideoTrack 之后调用了 `addTransceiver(direction)`
- **修复**：在 YangRtcWrapper 中新增 `addTransceiver()` 方法，在 videocallmanager.cpp 的两条初始化路径中添加调用
