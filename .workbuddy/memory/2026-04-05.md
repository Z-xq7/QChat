### 修复群聊登录后不显示的 Bug（AppendGroupList 与 slot_load_chat_thread 判重冲突）

而 `slot_load_chat_thread` 在处理群聊时用 `IsGroupChat()` 判重，一旦 true 就 `continue` 跳过，**UI item 永远不会创建**。

**修复方案（chatdialog.cpp `slot_load_chat_thread` 中）**：
- 改用 `_chat_thread_items.contains(thread_id)` 来判重（UI 是否已存在）
- 如果 `_group_chat_map` 里已有完整数据，直接复用（含公告、图标）

### 修复群聊消息不显示 + 消息类型标记全面修复

**根因**：群聊页面中消息不显示是两个独立问题叠加的结果。

**问题2：ChatFormType 全部写死 PRIVATE** — 整个 tcpmgr.cpp 中所有创建 ChatData 的地方都用 `ChatFormType::PRIVATE`，群聊消息的类型标记错误。
- **修复策略**：对于有服务端返回 `is_group` 字段的场景（`ID_TEXT_CHAT_MSG_RSP`、`ID_NOTIFY_TEXT_CHAT_MSG_REQ`），直接用 `jsonObj["is_group"].toBool()`；对于历史消息加载和 RSP 场景（`ID_LOAD_CHAT_MSG_RSP`、`ID_FILE_CHAT_MSG_RSP`、`ID_IMG_CHAT_MSG_RSP`、`ID_NOTIFY_FILE_CHAT_MSG_REQ`、`ID_NOTIFY_IMG_CHAT_MSG_REQ`、`CreatePlaceholderImgMsgL`），用 `UserMgr::GetInstance()->IsGroupChat(thread_id)` 本地判断
- **服务端同步**：`ChatServer::DealChatTextMsg` 和 `ChatServer2::DealChatTextMsg` 的 `rtvalue` 中加入 `is_group` 字段

**问题3：群聊中非好友消息不显示** — `ChatPage::AppendChatMsg` 的 Others 分支用 `GetFriendById` 查找发送者，找不到直接 return。群成员不一定是好友，导致群聊中他人发的消息丢失。

- **Self 分支**：群聊中自己发的消息也加了 `showUserName(true)`

**已知遗留问题**：群聊文件/图片消息上传完成后，ResourceServer 只通知 `chat_msg->recv_id` 对应的那一个用户（私聊逻辑），群聊中其他成员收不到文件/图片通知。这需要 ResourceServer 做更大的改造（遍历群成员逐一通知）。

### 修复好友在线时切换联系人列表卡死

**现象**：如果好友在线，登录后切换到联系人列表（side_contact_lb）UI 卡死；好友全部离线时正常。

**根因分析**：`SetFriendOnlineStatusBatch` 对每个在线好友逐个 emit `sig_friend_status_changed`，每个信号同步调用 `UpdateContactOnlineStatus` 遍历整个 `con_user_list`（O(n²) 复杂度），同时外层 lambda 又调 `UpdateAllContactOnlineStatus` 再次全量遍历。双重遍历 + 大量 show/hide/setText UI 操作可能导致布局系统过载。

**修复方案**：
1. `UserMgr::SetFriendOnlineStatusBatch`：不再逐个 emit `sig_friend_status_changed`，改为只 emit 一次 `sig_friend_online_status_batch()` 信号
2. `UserMgr` 新增信号 `sig_friend_online_status_batch()`
3. `ChatDialog`：连接 `sig_friend_online_status_batch` → `UpdateAllContactOnlineStatus()`（只遍历一次列表）
4. `ChatPage`：连接 `sig_friend_online_status_batch` → 刷新当前页面的在线状态显示
5. 单个好友上/下线通知仍然使用 `sig_friend_status_changed`（只影响一个好友，开销小）

### 修复已读回执三个关键 Bug

**Bug 1（卡死）**：好友上线后卡住一段时间才加载完成
- **根因**：`SetSelectChatItem()` 调用 `setCurrentRow(0)` 触发了 `itemClicked` 信号 → `slot_item_clicked` → 发送 `ID_MARK_MSG_READ_REQ` → 服务端 `GetFriendList` 查 MySQL 遍历所有好友 + `GetUnreadCounts` SQL 缺少 `(recv_id, status)` 索引全表扫描
- **修复**：
  1. `SetSelectChatItem` 用 `blockSignals(true)` 防止触发 `itemClicked`
  2. `SetSelectChatItem` 移除了直接清零未读计数的逻辑（这应该由 `slot_item_clicked` 统一处理）
  3. `chat_message` 表新增 `idx_recv_status(recv_id, status)` 和 `idx_thread_recv_status(thread_id, recv_id, status)` 索引

**Bug 2（未读数被清空）**：不管有没有打开聊天页面，未读角标都被清空
- **根因**：同 Bug 1，`SetSelectChatItem` 触发 `slot_item_clicked` 在 `GetUnreadCounts` 响应到达前就清零了第一个项的角标
- **修复**：同 Bug 1，blockSignals 后 `SetSelectChatItem` 不再触发 `slot_item_clicked`

**Bug 3（已读通知不生效）**：接收方读消息后发送方状态不变
- **根因**：`MarkMsgReadHandler` 遍历**所有好友**发 `ID_NOTIFY_MSG_READ`，不精确；而 `slot_item_clicked` 被 `SetSelectChatItem` 误触发后发了多余的 MarkMsgRead 请求
- **修复**：
  1. 服务端 `MarkMsgReadHandler` 不再遍历好友列表，改为 `GetPrivateChatPeer` 精确查找对方 uid 并直接通知
  2. 新增 `MysqlDao::GetPrivateChatPeer(thread_id, self_uid, peer_uid)` 方法
  3. ChatServer + ChatServer2 同步修改（MysqlDao/MysqlMgr/LogicSystem）

### 修复 MySQL 连接池 autocommit 残留导致锁表

**现象**：Navicat 中执行 DELETE 一条记录都卡死，`SHOW PROCESSLIST` 显示 12 个 Sleep 连接（来自 ChatServer/ChatServer2 连接池）。

**根因**：连接池中某些方法（如 CreateGroupChat）手动 `setAutoCommit(false)` 后，异常分支只 rollback 但**没有恢复 autocommit=true**。连接归还到池后，下一个复用该连接的操作在 autocommit=false 模式下执行，未显式 commit 的改动持有行锁/表锁，导致其他操作阻塞。

**修复**：`MySqlPool::returnConnection()` 归还前检查并强制恢复 autocommit=true + rollback 残留事务。ChatServer + ChatServer2 同步修改。

**经验教训**：连接池的 returnConnection 是最后防线，必须确保归还的连接处于干净状态（autocommit=true、无残留事务）。
